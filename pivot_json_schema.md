# Pivot Extractor JSON — Schema Reference

Generated by `pivot_extractor_v2.py`. This file describes every key, its type, allowed values, and how it should be interpreted when building DAX queries, Power BI models, or other downstream consumers.

---

## Top-level structure

```json
{
  "meta":               { ... },
  "master_sheet_info":  { ... },
  "pivots":             [ { ... }, ... ]
}
```

| Key | Type | Description |
|-----|------|-------------|
| `meta` | object | Extraction run metadata |
| `master_sheet_info` | object | Column inventory of the primary data table |
| `pivots` | array of objects | One entry per pivot table found in the workbook |

---

## `meta`

Describes the extraction run. Use this to verify the file and settings that produced the JSON before parsing `pivots`.

```json
{
  "source_file":  "sales.xlsx",
  "source_path":  "/data/sales.xlsx",
  "master_sheet": "Sales Data",
  "header_row":   1,
  "extracted_at": "2026-03-11T11:20:34.959593+00:00",
  "pivot_count":  3,
  "successful":   3,
  "failed":       0
}
```

| Key | Type | Description |
|-----|------|-------------|
| `source_file` | string | Filename only (no path) of the source `.xlsx` |
| `source_path` | string | Full path as passed to the extractor |
| `master_sheet` | string | Name of the sheet designated as the data source |
| `header_row` | integer | 1-based row number in `master_sheet` where column headers live. Rows above this are ignored. Default `1`. |
| `extracted_at` | ISO 8601 string | UTC timestamp of the extraction |
| `pivot_count` | integer | Total pivot tables found (successful + failed) |
| `successful` | integer | Pivot tables parsed without errors |
| `failed` | integer | Pivot tables that raised a parse error (see `pivots[].error`) |

## `master_sheet_info`

Column inventory read directly from the master data sheet. This is used to validate that pivot field names resolve to real columns in the data source.

```json
{
  "sheet":      "Sales Data",
  "path":       "xl/worksheets/sheet1.xml",
  "range":      "A1:I361",
  "header_row": 1,
  "row_count":  360,
  "col_count":  9,
  "columns": [
    {
      "index":         0,
      "name":          "Date",
      "sample_value":  "44934.0",
      "inferred_type": "number"
    },
    ...
  ]
}
```

| Key | Type | Description |
|-----|------|-------------|
| `sheet` | string | Sheet name |
| `path` | string | Internal zip path of the sheet XML (diagnostic use) |
| `range` | string \| null | Used range as declared in the sheet's `<dimension>` element, e.g. `"A1:I361"`. `null` if not present |
| `header_row` | integer | Same as `meta.header_row` |
| `row_count` | integer | Number of data rows below the header row |
| `col_count` | integer | Number of non-null columns found in the header row |
| `columns` | array | One entry per header column (see below) |

### `master_sheet_info.columns[]`

| Key | Type | Description |
|-----|------|-------------|
| `index` | integer | 0-based column index within the sheet |
| `name` | string | Header cell value — this is the canonical column name used for linking |
| `sample_value` | string \| null | Stringified value from the first data row (row `header_row + 1`), for type hints |
| `inferred_type` | string | `"number"`, `"boolean"`, `"text"`, or `"unknown"` — inferred from `sample_value`'s Python type after Excel parsing. Note: dates stored as serial numbers will appear as `"number"` |

> **Note on date columns:** Excel stores dates as floating-point serial numbers internally. A `Date` column with `inferred_type: "number"` and a `sample_value` like `"44934.0"` is a date. Use the pivot cache field's `data_type: "date"` (in `pivots[].fields`) as the authoritative type signal.

## `pivots[]`

Each element represents one PivotTable extracted from the workbook. On parse failure, only `id`, `host_sheet`, `pivot_file`, `error`, and `trace` are present.

### Top-level pivot fields

```json
{
  "id":               "PT_Sales_Summary_1",
  "name":             "SalesByRegion",
  "host_sheet":       "Dashboard",
  "cache_id":         0,
  "pivot_file":       "xl/pivotTables/pivotTable1.xml",
  "cache_source":     { ... },
  "location":         { ... },
  "row_grand_total":  true,
  "col_grand_total":  true,
  "data_caption":     "Values",
  "compact_mode":     false,
  "outline_mode":     false,
  "style":            "PivotStyleMedium9",
  "fields":           { ... },
  "rows":             [ ... ],
  "columns":          [ ... ],
  "filters":          [ ... ],
  "values":           [ ... ],
  "pivot_filters":    [ ... ],
  "calculated_fields":[ ... ],
  "master_column_links": { ... },
  "warnings":         [ ... ],
  "dax":              "-- ===...\nDEFINE\n..."
}
```

| Key | Type | Description |
|-----|------|-------------|
| `id` | string | Synthetic unique identifier in the form `PT_{SheetName}_{N}` |
| `name` | string | The pivot table's name as set in Excel (shown in PivotTable Tools) |
| `host_sheet` | string | Name of the worksheet where the pivot table is rendered |
| `cache_id` | integer \| null | Excel internal cache ID linking this pivot to its data cache |
| `pivot_file` | string | Internal zip path of the pivot XML (diagnostic use) |
| `cache_source` | object | Where the pivot's data comes from — see below |
| `location` | object \| null | Cell range and layout offsets of the rendered table |
| `row_grand_total` | boolean | Whether a grand total row is shown at the bottom |
| `col_grand_total` | boolean | Whether a grand total column is shown on the right |
| `data_caption` | string | The label shown in the Values drop-zone (often `"Values"` or `""`) |
| `compact_mode` | boolean | If `true`, multiple row fields share one column in the rendered table |
| `outline_mode` | boolean | If `true`, the pivot uses outline layout |
| `style` | string \| null | Excel table style name applied to the pivot |
| `fields` | object | Map of `fieldName → fieldInfo` for every field in the cache — see `fields` section |
| `rows` | array | Ordered list of fields placed on the Row axis — see `rows[]` |
| `columns` | array | Ordered list of fields placed on the Column axis — see `columns[]` |
| `filters` | array | Page/report filter fields — see `filters[]` |
| `values` | array | Value (data) field definitions — see `values[]` |
| `pivot_filters` | array | Label, value, and Top 10 filters applied inside the pivot — see `pivot_filters[]` |
| `calculated_fields` | array | Calculated fields defined in this pivot — see `calculated_fields[]` |
| `master_column_links` | object | Resolution of each field name against `master_sheet_info.columns` — see below |
| `warnings` | array | Non-fatal issues noted during parsing (strings) |
| `dax` | string | Generated DAX `DEFINE … EVALUATE` block ready for Power BI / DAX Studio |
| `error` | string | *(only on failed pivots)* Exception message |
| `trace` | string | *(only on failed pivots)* Full Python traceback |

### `cache_source`

Describes where the pivot's data originates.

```json
{
  "type":  "worksheet",
  "sheet": "Sales Data",
  "range": "A1:I361",
  "name":  null
}
```

| Key | Type | Description |
|-----|------|-------------|
| `type` | string | `"worksheet"` for a sheet range source; other values (e.g. `"external"`, `"consolidation"`) are passed through as-is |
| `sheet` | string \| null | Source sheet name. Should match `meta.master_sheet` for standard single-source pivots |
| `range` | string \| null | Cell range of the source data, e.g. `"A1:I361"` |
| `name` | string \| null | Named table or range name if the source is a defined table, otherwise `null` |

---

### `location`

The rendered position of the pivot table on its host sheet.

```json
{
  "ref":              "A1:M8",
  "first_header_row": 0,
  "first_data_row":   2,
  "first_data_col":   1
}
```

| Key | Type | Description |
|-----|------|-------------|
| `ref` | string | Full cell range occupied by the rendered pivot (includes headers and data) |
| `first_header_row` | integer | 0-based offset of the first header row within `ref` |
| `first_data_row` | integer | 0-based offset of the first data row within `ref` |
| `first_data_col` | integer | 0-based offset of the first data column within `ref` (columns to the left are row-label columns) |

---

### `fields`

A flat object keyed by field name. Contains **every** field from the pivot cache — not just the ones placed on an axis. Use `placement` to determine role.

```json
{
  "Region": {
    "index":             1,
    "data_type":         "text",
    "num_format":        "General",
    "is_calculated":     false,
    "formula":           null,
    "grouping":          null,
    "is_database_field": true,
    "placement":         "axisRow",
    "subtotals":         ["sum"],
    "hidden_items":      [],
    "distinct_values":   [
      { "type": "string", "value": "North" },
      { "type": "string", "value": "South" }
    ]
  }
}
```

| Key | Type | Description |
|-----|------|-------------|
| `index` | integer | 0-based position in the cache field list. Used to cross-reference `rows[].index`, `columns[].index`, and `values[].field_index` |
| `data_type` | string | `"text"`, `"number"`, `"date"`, `"boolean"`, `"mixed"`, `"grouped"`, or `null`. `"mixed"` means the column contains both strings and numbers. `"grouped"` means the field is a grouping overlay (e.g. date grouped by Year) |
| `num_format` | string \| null | Excel number format string, e.g. `"yyyy-mm-dd"`, `"#,##0.00"`, `"General"` |
| `is_calculated` | boolean | `true` if this is a calculated field (defined by a formula, not a raw data column) |
| `formula` | string \| null | The Excel calculated field formula, e.g. `"= Revenue - COGS"`. Present when `is_calculated` is `true` |
| `grouping` | object \| null | Date/number grouping definition — see `grouping` section below |
| `is_database_field` | boolean | `true` for columns that map directly to source data. `false` for calculated fields injected into the cache |
| `placement` | string | Where this field is used in the pivot. One of: `"axisRow"`, `"axisCol"`, `"axisPage"`, `"values"`, `"unused"` |
| `subtotals` | array of strings | Subtotal functions enabled for this field, e.g. `["sum"]`, `["sum", "count"]`. Empty array means default subtotal only |
| `hidden_items` | array | List of member values that have been hidden/filtered out within the field. Items are the raw values (strings, numbers). Empty array if nothing is hidden |
| `distinct_values` | array | Up to 300 distinct member values from the pivot cache. Each entry is `{ "type": ..., "value": ... }` — see distinct value types below. **This array can be large (300+ entries for high-cardinality fields). Do not iterate it to build pivot table row/column combinations.** |

#### `distinct_values[].type` values

| Value | Meaning |
|-------|---------|
| `"string"` | Text member |
| `"number"` | Numeric member |
| `"date"` | ISO 8601 date string, e.g. `"2023-01-08T00:00:00Z"` |
| `"bool"` | Boolean |
| `"blank"` | Empty / null cell |
| `"error"` | Excel error value (e.g. `#N/A`) |

#### `grouping` (when not `null`)

Present when a field has Excel grouping applied (e.g. a Date field grouped by Year+Month, or a numeric field bucketed into ranges).

**Range grouping** (dates, numeric ranges):
```json
{
  "type":        "range",
  "group_by":    "months",
  "auto_start":  true,
  "auto_end":    true,
  "start_value": "2023-01-01T00:00:00Z",
  "end_value":   "2024-12-31T00:00:00Z",
  "interval":    null,
  "base_field":  0,
  "buckets":     ["Jan", "Feb", "Mar", ...]
}
```

| Key | Description |
|-----|-------------|
| `type` | Always `"range"` for this variant |
| `group_by` | Grouping granularity: `"years"`, `"quarters"`, `"months"`, `"days"`, `"hours"`, `"minutes"`, `"seconds"`, or a numeric interval label |
| `auto_start` / `auto_end` | Whether the start/end boundaries are auto-detected from the data |
| `start_value` / `end_value` | Boundary values (ISO date string for date fields, numeric string for number fields), or `null` if auto |
| `interval` | Step size for numeric bucketing (e.g. `"1000"`), or `null` for date groupings |
| `base_field` | Index of the underlying ungrouped cache field this grouping is derived from |
| `buckets` | Ordered list of bucket label strings as they appear in the pivot |

**Discrete grouping** (manual custom groups):
```json
{
  "type":     "discrete",
  "buckets":  ["Group A", "Group B"],
  "mappings": [0, 1, 0, 1, 0]
}
```

| Key | Description |
|-----|-------------|
| `type` | Always `"discrete"` for manually defined groups |
| `buckets` | Group label names |
| `mappings` | Per-item mapping: index into `buckets` for each distinct value in the base field |

> **DAX implication for grouped fields:** Excel date groupings have no direct equivalent in Power BI. The `dax` block emits `YEAR()`, `MONTH()`, or `QUARTER()` wrappers with a comment recommending a dedicated Date table instead.

### `rows[]`

Ordered array of fields placed on the **Row** axis of the pivot. Order determines nesting (first entry is outermost).

Each entry is one of two types:

**Regular field:**
```json
{ "type": "field", "field": "Region", "index": 1 }
```

**Values placeholder** (appears when the Values area is shown as a row rather than a column — rare):
```json
{ "type": "values_header", "field": "__VALUES__" }
```

| Key | Type | Description |
|-----|------|-------------|
| `type` | string | `"field"` or `"values_header"` |
| `field` | string | Field name. For `"values_header"` this is always `"__VALUES__"` — skip it when building GROUP BY clauses |
| `index` | integer | Cache field index (matches `fields[name].index`) |

---

### `columns[]`

Ordered array of fields placed on the **Column** axis. Structure identical to `rows[]` with one addition:

**Regular field** (with resolved members):
```json
{
  "type":    "field",
  "field":   "Sales_Rep",
  "index":   8,
  "members": ["Eve", "Alice", "Bob", "Dave", "Carol"]
}
```

| Key | Type | Description |
|-----|------|-------------|
| `type` | string | `"field"` or `"values_header"` |
| `field` | string | Field name |
| `index` | integer | Cache field index |
| `members` | array \| absent | **Only present on `"field"` entries.** The ordered list of distinct member values for this column field, as resolved from the pivot cache. These are the column header labels visible in Excel (e.g. the individual sales rep names). Use this list — not `fields[name].distinct_values` — when enumerating column headers, as it respects the cache's item ordering and any hidden-item exclusions. |

> **`__VALUES__` sentinel:** When the pivot has multiple value fields and they are spread across columns, a `{ "type": "values_header", "field": "__VALUES__" }` entry appears in `columns[]` to mark where the value field sub-headers are positioned. Ignore it for DAX GROUP BY — it is a layout marker only.

---

### `filters[]`

Report/page filter fields (the "Filters" zone at the top of a PivotTable). These apply globally to the entire pivot.

```json
{
  "field":          "Year",
  "field_index":    0,
  "selected_item":  "2024",
  "selected_index": 2,
  "show_all":       false
}
```

| Key | Type | Description |
|-----|------|-------------|
| `field` | string | Field name |
| `field_index` | integer | Cache field index |
| `selected_item` | any \| null | The single selected filter value (string, number, etc.). `null` if all items are shown |
| `selected_index` | integer \| null | 0-based index into `fields[name].distinct_values` of the selected item |
| `show_all` | boolean | `true` when no filter is active (all items shown). When `false`, `selected_item` holds the active filter value |

> **DAX implication:** When `show_all` is `false`, the generated DAX wraps aggregations in `CALCULATE(..., 'Table'[Field] = "value")`.

---

### `values[]`

Value (measure) fields placed in the data area of the pivot. Each defines what is aggregated and how it is displayed.

```json
{
  "display_name":       "SUM of Revenue",
  "source_field":       "Revenue",
  "field_index":        6,
  "aggregation":        "sum",
  "aggregation_label":  "SUM",
  "show_data_as":       "percentOfTotal",
  "show_data_as_label": "% of grand total",
  "base_field":         null,
  "base_item":          null,
  "num_format":         "#,##0.00",
  "is_calculated":      false,
  "formula":            null
}
```

| Key | Type | Description |
|-----|------|-------------|
| `display_name` | string \| null | The label shown in the pivot header, e.g. `"Sum of Revenue"`. Use this as the DAX measure column name |
| `source_field` | string | The underlying cache field being aggregated. For calculated value fields, this may be `"__VALUES__"` |
| `field_index` | integer | Cache field index of `source_field`. `-2` indicates the values placeholder |
| `aggregation` | string | Excel aggregation function key. See aggregation values table below |
| `aggregation_label` | string | Human-readable label for `aggregation`, e.g. `"SUM"`, `"AVERAGE"` |
| `show_data_as` | string | How the aggregated value is transformed for display. See `show_data_as` values table below |
| `show_data_as_label` | string | Human-readable label for `show_data_as` |
| `base_field` | string \| null | For difference/percent/running-total transforms: the field name used as the reference axis |
| `base_item` | any \| null | For difference/percent transforms: the specific member value of `base_field` used as the reference point |
| `num_format` | string \| null | Excel number format to apply to the values, e.g. `"0.00%"`, `"#,##0"` |
| `is_calculated` | boolean | `true` if this value field is backed by a calculated field formula |
| `formula` | string \| null | The Excel calculated field formula when `is_calculated` is `true` |

#### `aggregation` values

| Value | Excel function | DAX equivalent |
|-------|---------------|----------------|
| `"sum"` | Sum | `SUM(...)` |
| `"count"` | Count | `COUNTA(...)` |
| `"average"` | Average | `AVERAGE(...)` |
| `"max"` | Max | `MAX(...)` |
| `"min"` | Min | `MIN(...)` |
| `"product"` | Product | `PRODUCTX(...)` |
| `"countNums"` | Count Numbers | `COUNT(...)` |
| `"stdDev"` | StdDev (sample) | `STDEV.S(...)` |
| `"stdDevP"` | StdDev (population) | `STDEV.P(...)` |
| `"var"` | Variance (sample) | `VAR.S(...)` |
| `"varP"` | Variance (population) | `VAR.P(...)` |

#### `show_data_as` values

| Value | Meaning | DAX pattern |
|-------|---------|-------------|
| `"normal"` | Raw aggregated value | Direct aggregation |
| `"percentOfTotal"` | % of grand total | `DIVIDE(value, CALCULATE(value, ALL(table)))` |
| `"percentOfRow"` | % of row total | `DIVIDE(value, CALCULATE(value, ALLSELECTED()))` |
| `"percentOfCol"` | % of column total | `DIVIDE(value, CALCULATE(value, ALLSELECTED()))` |
| `"percent"` | % of base item | `DIVIDE(value, CALCULATE(value, field = base_item))` |
| `"difference"` | Difference from base item | `value - CALCULATE(value, field = base_item)` |
| `"percentDiff"` | % difference from base item | `DIVIDE(value - base, base)` |
| `"runTotal"` | Running total | `CALCULATE(value, FILTER(ALLSELECTED(field), field <= MAX(field)))` |
| `"index"` | Index | `(cell × grand_total) / (row_total × col_total)` |
| `"percentOfParentRow"` | % of parent row total | Requires hierarchy context |
| `"percentOfParentCol"` | % of parent col total | Requires hierarchy context |
| `"percentOfParent"` | % of parent | Requires hierarchy context |
| `"rankAscending"` | Rank ascending | `RANKX(...)` ascending |
| `"rankDescending"` | Rank descending | `RANKX(...)` descending |

> When `show_data_as` is not `"normal"`, the `base_field` and `base_item` keys provide the reference axis. Both may be `null` for transforms that don't require a base (e.g. `percentOfTotal`).

### `pivot_filters[]`

In-pivot filters — label filters, value filters, and Top 10 filters applied to row/column fields directly within the pivot (distinct from the page filters in `filters[]`).

```json
{
  "field":       "Product",
  "field_index": 3,
  "filter_type": "count",
  "eval_order":  null,
  "criteria":    {
    "type": "count",
    "conditions": [
      {
        "kind":    "top10",
        "column":  0,
        "top":     true,
        "percent": false,
        "val":     "5"
      }
    ]
  }
}
```

| Key | Type | Description |
|-----|------|-------------|
| `field` | string | Field being filtered |
| `field_index` | integer | Cache field index |
| `filter_type` | string | Excel filter type string (e.g. `"count"`, `"sum"`, `"captionEqual"`, `"valueGreaterThan"`) |
| `eval_order` | integer \| null | Evaluation order when multiple filters exist |
| `criteria` | object | Filter condition details — see below |

#### `criteria.conditions[]` — `kind` variants

**`"valueList"`** — filter to a specific set of members:
```json
{ "kind": "valueList", "column": 0, "values": ["Tablet", "Laptop"], "include_blanks": false }
```

**`"custom"`** — comparison filter (greater than, between, etc.):
```json
{
  "kind": "custom", "column": 0,
  "operator": "AND",
  "filters": [
    { "operator": "greaterThanOrEqual", "value": "1000" },
    { "operator": "lessThan",           "value": "5000" }
  ]
}
```
Comparison operators: `"equal"`, `"notEqual"`, `"greaterThan"`, `"greaterThanOrEqual"`, `"lessThan"`, `"lessThanOrEqual"`.

**`"top10"`** — Top/Bottom N filter:
```json
{ "kind": "top10", "column": 0, "top": true, "percent": false, "val": "5" }
```
`top: true` = Top N, `false` = Bottom N. `percent: true` = Top N%, `false` = Top N items.

**`"dynamic"`** — date-relative dynamic filter (e.g. "This Month", "Last Year"):
```json
{ "kind": "dynamic", "column": 0, "type": "thisMonth", "val": null }
```

### `calculated_fields[]`

Calculated fields defined within this pivot. These are formula-based virtual columns that do not exist in the source data.

```json
{
  "name":    "Profit",
  "formula": "= Revenue - COGS",
  "context": "value_field"
}
```

| Key | Type | Description |
|-----|------|-------------|
| `name` | string | Field name as shown in the pivot |
| `formula` | string | The Excel formula. Uses bare column names or `[ColumnName]` bracket syntax. Arithmetic operators and Excel functions are used as written |
| `context` | string | Always `"value_field"` in the current implementation — calculated fields appear in the Values zone |

> **DAX translation:** The generated `dax` string contains a translated version of each formula with Excel-style `[Field]` references replaced by `'Table'[Field]` DAX references. Common Excel functions are mapped to DAX equivalents (e.g. `TEXT` → `FORMAT`). Functions with no direct DAX equivalent are preserved with a `-- TODO` comment.

### `master_column_links`

Resolution map confirming which pivot cache fields correspond to actual columns in the master sheet. Used to validate that the DAX `'Table'[Field]` references will resolve.

```json
{
  "Revenue": {
    "matched_master_column": "Revenue",
    "is_calculated":         false,
    "linked":                true
  },
  "Profit": {
    "matched_master_column": null,
    "is_calculated":         true,
    "linked":                true
  },
  "UnknownField": {
    "matched_master_column": null,
    "is_calculated":         false,
    "linked":                false
  }
}
```

Keys are field names from the pivot cache. For each:

| Key | Type | Description |
|-----|------|-------------|
| `matched_master_column` | string \| null | The exact column name from `master_sheet_info.columns` that this field resolved to (case-insensitive match). `null` if no match was found |
| `is_calculated` | boolean | Whether this field is a calculated field (does not need a source column match) |
| `linked` | boolean | `true` if the field is either matched to a master column OR is a calculated field. `false` means the field name exists in the cache but has no corresponding column in the master sheet — the generated DAX references for this field will be unresolved |

> Fields with `linked: false` indicate a mismatch between the pivot cache and the master sheet. Common causes: the pivot uses a different data source than `master_sheet`, the column was renamed, or the workbook contains only a partial data extract.

### `dax`

A fully formed DAX string ready for use in Power BI Desktop or DAX Studio. The block uses `DEFINE` / `EVALUATE` syntax.

Structure:
```
-- Header comment (pivot name, source, row/col/value fields, calc field translations)

DEFINE
    -- ── Value field N: <display_name>
    --    Aggregation : SUM
    --    ShowDataAs  : Raw value
    --    Source field: Revenue
    VAR _V1_SUM_of_Revenue = SUM('Sales Data'[Revenue])

    VAR _V2_... = ...

EVALUATE

SUMMARIZECOLUMNS(
    'Sales Data'[Region],
    'Sales Data'[Sales_Rep],
    "SUM of Revenue", _V1_SUM_of_Revenue,
    ...
)

-- Grand total row equivalent:
EVALUATE ROW(
    "SUM of Revenue (Grand Total)",
    CALCULATE(_V1_SUM_of_Revenue, ALL('Sales Data')),
    ...
)
```

- Each value field becomes a `VAR` in the `DEFINE` block
- Row and column fields become GROUP BY columns in `SUMMARIZECOLUMNS`
- Page filters become `CALCULATE` filter arguments inside each VAR
- `showDataAs` transforms wrap the base aggregation with the appropriate DAX pattern
- The grand total block is only emitted when `row_grand_total` or `col_grand_total` is `true`
- Calculated fields include both the original Excel formula as a comment and the translated DAX expression
- Fields with `-- TODO` comments require manual review before the query will run correctly

## Parsing guidance for AI consumers

1. **Start with `meta`** to confirm `master_sheet` and `header_row` match your expectations before reading any pivot data.

2. **Use `master_sheet_info.columns`** as the authoritative column list for the Power BI table. Field names in the DAX are derived from these names.

3. **Iterate `pivots[]`** and skip any entry that has an `"error"` key — these are unparseable and have no `rows`, `columns`, or `values`.

4. **To reconstruct a pivot's axis structure:**
   - Row GROUP BY fields: `rows[]` entries where `field != "__VALUES__"`, in order
   - Column GROUP BY fields: `columns[]` entries where `field != "__VALUES__"`, in order
   - Column header values: `columns[].members` (not `fields[name].distinct_values`)
   - Measure definitions: `values[]`, one DAX measure per entry

5. **Do not expand `fields[name].distinct_values`** for column/row enumeration — it is a raw cache dump (up to 300 items, including high-cardinality fields like dates) and will cause memory issues. Use `columns[].members` for column members and `filters[].selected_item` for filter state.

6. **`master_column_links`** tells you immediately which fields are safe to reference as `'Table'[Field]` in DAX (`linked: true`) and which need investigation (`linked: false`).

7. **Calculated fields** appear in both `calculated_fields[]` (with the formula) and in `fields[name]` (with `is_calculated: true`). The translated DAX formula is embedded in the `dax` string as a comment alongside the VAR definition.

8. **The `dax` field** is the primary output for Power BI work. It is self-contained: copy it directly into DAX Studio's query editor or a Power BI calculated table expression and it will run against a table named after `meta.master_sheet`.
